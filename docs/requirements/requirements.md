## 1. 项目背景

### 1.1 现有项目概况

`wow_wrapper` 是一个专为魔兽世界私服（基于 AzerothCore 框架）设计的数据查询与展示 Web 服务，通过连接私服数据库，提供各类游戏数据的实时查询和可视化展示。

**现有功能模块：**

| 模块 | 功能说明 | 对应 API |
|------|----------|----------|
| 在线统计 | 实时显示联盟/部落在线人数及在线玩家列表 | `count_online`, `online_player` |
| 综合排行 | 坐骑、成就、荣誉、金币、游戏时长多维度排行 | `top_mount`, `top_achievement`, `top_honor`, `top_gold`, `top_playtime` |
| 硬核模式 | 硬核挑战通关(60/70/80级)/失败/进行中的玩家榜单 | `hardcore_completed_*`, `hardcore_fail`, `hardcore_incompleted` |
| 最近成就 | 展示玩家近期获得的成就 | `recent_achieve` |
| 封禁列表 | 查看被封禁的玩家记录 | `recent_ban_list` |
| 角色信息 | 查询角色基础信息及装备栏 | `CharInfo`, `CharItem` |

**现有技术栈：**

| 层级 | 技术 |
|------|------|
| 容器化 | Docker + Docker Compose |
| Web 服务器 | Nginx |
| 后端 | PHP 8.0 + Composer + Medoo ORM |
| 缓存 | Redis |
| 数据库 | MySQL（AzerothCore） |
| 前端 | jQuery + Bootstrap 5 + DataTables |
| 进程管理 | Supervisor |

### 1.2 现有项目痛点

1. **技术栈陈旧**：后端使用 PHP，维护成本高，生态活跃度下降，与现代云服务兼容性差。
2. **前后端技术不一致**：后端 PHP，前端原生 JS，语言栈割裂，无法共享类型定义和工具链。
3. **前端技术落后**：jQuery + DataTables 的方案在 2026 年已属陈旧，组件化程度低，可维护性差。
4. **资源管理原始**：部分静态资源（如 `app/src/csv/spell.csv`）依赖手动在服务器上拷贝操作，无法纳入版本管理和 CI/CD 流程。
5. **部署困难**：无法做到在腾讯云上通过云函数（SCF）或容器镜像快速部署，运维成本高。

---

## 2. 项目目标

将 `wow_wrapper` 的现有功能，使用现代化的技术栈重新实现，并按照行业标准规范组织项目，最终达到以下目标：

1. **技术栈现代化**：使用当前行业流行的全栈技术，前后端统一技术语言。
2. **工程规范标准化**：项目目录、代码结构、开发流程符合行业最佳实践。
3. **CI/CD 自动化**：代码提交后自动构建、测试、打包镜像，并支持在腾讯云上快速部署。

## 3. 功能需求

### 3.1 功能范围（与 legacy 项目对齐）

新项目需完整覆盖 `wow_wrapper` 的全部现有功能，保持 API 接口能力等价。

#### 3.1.1 在线统计模块

- **在线人数统计**：返回当前在线总人数、联盟人数、部落人数。
- **在线玩家列表**：返回所有在线玩家的 GUID、名称、种族、职业、等级、性别、阵营信息。

#### 3.1.2 综合排行模块

- **坐骑排行**：按坐骑持有数量降序排列，返回前 100 名玩家信息及坐骑列表。
- **成就排行**：按成就点数降序排列，返回前 200 名玩家信息。
- **荣誉排行**：按荣誉点数降序排列，返回前 200 名玩家信息。
- **金币排行**：按持有金币数降序排列，返回前 200 名玩家信息。
- **游戏时长排行**：按游戏总时长降序排列，返回前 200 名玩家信息。

#### 3.1.3 硬核模式模块

- **硬核通关榜（60级）**：返回完成 60 级硬核挑战的玩家列表。
- **硬核通关榜（70级）**：返回完成 70 级硬核挑战的玩家列表。
- **硬核通关榜（80级）**：返回完成 80 级硬核挑战的玩家列表。
- **硬核失败榜**：返回硬核挑战失败的玩家列表（最近 500 条）。
- **硬核进行中**：返回正在进行硬核挑战且未达到满级的玩家列表。

#### 3.1.4 最近成就模块

- 返回最近获得成就的玩家列表（最近 50 条），包含成就名称和获取时间。

#### 3.1.5 封禁列表模块

- 返回当前处于封禁状态的玩家列表（最近 200 条），包含用户名、封禁时间、解封时间、封禁原因。

#### 3.1.6 角色信息模块

- **角色基础信息**：通过角色名称查询等级、种族、职业、生命值、能量值、游戏时长、荣誉等级、竞技场点数、总击杀数、完成任务数、获得成就数。
- **角色装备信息**：通过角色名称查询 19 个装备栏位的图标信息。

### 3.2 数据需求

- **数据来源**：直接连接 AzerothCore 私服数据库（`acore_auth`、`acore_characters`、`acore_world`）。
- **数据时效性**：热点数据（在线统计、排行等）通过缓存机制降低数据库压力，缓存有效期可配置。
- **坐骑数据**：`spell.csv` 中的坐骑定义数据需纳入项目资源管理，不再依赖手动服务器拷贝。

## 4. 非功能需求

### 4.1 技术栈需求

| 层级 | 要求 |
|------|------|
| 编程语言 | 前后端统一使用 **TypeScript** 或现代语言（如 Go/Rust），优先考虑全栈一致性 |
| 后端框架 | 选用主流、轻量、适合云函数/容器部署的框架（如 Express/Fastify/NestJS/Hono/Fiber 等） |
| 前端框架 | 选用现代组件化框架（如 React/Vue/Svelte）配合 UI 组件库 |
| 数据库访问 | 使用现代 ORM/Query Builder（如 Prisma/Drizzle/TypeORM/GORM） |
| 缓存 | Redis，封装统一的缓存层 |
| 容器化 | Docker 多阶段构建，产出轻量生产镜像 |

### 4.2 项目结构需求

- 按照行业标准规范组织项目目录，前后端代码分离且结构清晰。
- 配置文件与环境变量分离，支持多环境（开发/测试/生产）。
- 静态资源纳入版本管理，构建时自动打包到产物中。
- 代码需具备完整的类型定义（TypeScript）或强类型保证。

### 4.3 质量保障需求

- 单元测试覆盖核心业务逻辑。
- 集成测试覆盖数据库查询和 API 接口。
- 代码风格统一（ESLint/Prettier/Biome 等）。
- 提交前自动执行 lint 和类型检查（Git Hooks / CI）。

### 4.4 部署与配置需求

#### 4.4.1 环境变量配置

- **全量配置通过 `.env` 管理**：所有敏感信息（数据库连接、Redis 连接、API Key、JWT Secret 等）均通过 `.env` 环境变量文件配置，代码中不硬编码任何环境相关的值。
- **多环境支持**：项目需提供 `.env.example` 模板文件，开发/测试/生产环境通过不同 `.env` 文件或环境变量注入切换。
- **云函数友好**：配置项设计需兼容腾讯云云函数（SCF）的环境变量配置方式，支持在控制台直接配置 `DB_HOST`、`REDIS_HOST`、`API_KEY` 等变量，服务启动时自动读取，无需修改代码或镜像。
- **配置校验**：服务启动时对必需的环境变量进行校验，缺失关键配置时给出明确错误提示并拒绝启动。

#### 4.4.2 CI/CD 与部署

- **CI/CD 流水线**：代码提交后自动触发构建、测试、镜像打包和推送。
- **腾讯云部署**：支持通过腾讯云容器镜像服务（TCR）推送镜像，并能在腾讯云云函数（SCF）或容器服务上快速部署运行。
- **独立部署**：acore-ranking 作为独立服务部署，不与 acore-api 耦合。两者共用 AzerothCore 数据库，但各自拥有独立的运行实例、镜像和配置生命周期。
- **配置外置**：数据库连接、API Key 等敏感信息通过环境变量注入，不写入镜像。

### 4.5 前端兼容性需求

#### 4.5.1 iframe 嵌入支持

- 所有展示页面需支持通过 `<iframe>` 嵌入到外部站点（如 WordPress 页面）中正常展示。
- 页面需正确设置响应头，允许跨域 iframe 嵌入（`X-Frame-Options` 和 `Content-Security-Policy` 配置）。
- 页面在 iframe 内部需自适应容器尺寸，不因 iframe 边界导致布局溢出或功能异常。
- 页面内所有跳转、Tab 切换、弹窗等交互行为在 iframe 环境中表现正常，不破坏父页面。

#### 4.5.2 移动端适配

- 所有展示页面需支持在移动设备（手机、平板）上正常查看和操作。
- 采用响应式布局设计，根据屏幕宽度自动调整表格、图表、导航等组件的展示方式。
- 移动端表格需支持横向滑动查看或切换为卡片式列表展示，避免列被挤压导致不可读。
- 触摸交互（点击、滑动）需流畅，按钮和链接的点击区域需符合移动端可用性标准。

### 4.6 日志需求

- **标准化日志输出**：系统所有日志统一通过结构化日志库输出，格式规范统一，避免直接使用 `console.log`。
- **日志分级**：按照行业标准进行日志分级，至少支持以下级别：
  - `ERROR` — 系统错误、异常堆栈，需立即关注
  - `WARN` — 潜在问题、非预期行为但不影响核心功能
  - `INFO` — 关键业务流程记录（如服务启动、请求处理）
  - `DEBUG` — 开发调试信息，生产环境可关闭
- **请求日志**：每个 HTTP 请求需记录访问日志，包含请求方法、路径、状态码、响应时间、客户端 IP 等关键信息。
- **错误日志**：异常和错误需记录完整堆栈、上下文参数，便于问题定位。
- **日志格式**：生产环境建议输出 JSON 格式日志，便于接入日志收集和分析平台；开发环境可输出可读性强的文本格式。
- **日志级别可配置**：通过环境变量控制日志输出级别，生产环境默认 `INFO`，开发环境默认 `DEBUG`。

### 4.7 开源治理需求

- **许可证**：项目采用 **MIT 许可证**，以最宽松的方式授权，允许自由使用、修改和分发。
- **开源仓库标准化**：
  - 仓库根目录包含规范的 `README.md`（项目介绍、快速开始、贡献指南）。
  - 包含 `LICENSE` 文件，明确标注 MIT 许可证。
  - 包含 `.gitignore` 文件，排除依赖、构建产物、敏感配置文件。
  - 包含 `CONTRIBUTING.md` 贡献指南。
  - 代码提交遵循 Conventional Commits 规范。
- **Issue / PR 模板**：配置 GitHub Issue 模板和 Pull Request 模板，规范社区协作流程。

    "data": []
  }

### 8.2 关联项目调研：acore-api

`acore-api` 是 AzerothCore 官方社区维护的一个开源 RESTful API 项目（[GitHub 仓库](https://github.com/azerothcore/acore-api)），基于 **NestJS + TypeScript + TypeORM** 构建，许可证为 MIT。

**核心功能**：
- 账号管理（注册、登录、JWT 鉴权）
- 角色数据查询（在线玩家、角色统计、竞技场队伍）
- 角色服务（改名、转阵营、解锁、物品恢复等，通过 SOAP 调用游戏服务器）
- 战场逃兵记录查询
- 世界数据库 / 网站数据库操作

**技术栈**：
- 后端框架: NestJS 7.x
- ORM: TypeORM (MySQL)
- 配置: dotenv 环境变量
- 安全: Helmet、express-rate-limit、CORS
- 测试: Jest

**对 acore-ranking 项目的借鉴价值**：

| 方面 | 借鉴内容 |
|------|----------|
| **技术选型验证** | 证明了 NestJS + TypeScript + TypeORM 这套技术栈在 AzerothCore 生态中完全可行，且已有成熟实践 |
| **数据库实体参考** | `acore-api` 已定义了 `characters`、`auth`、`world` 等数据库的完整 TypeORM Entity，可直接复用或作为参考，避免从零定义 Schema |
| **多数据库连接** | 展示了如何同时连接 `auth`、`characters`、`world` 多个数据库，acore-ranking 可直接参考其实现模式 |
| **环境变量配置** | 所有数据库连接、SOAP、JWT、邮件等配置均通过 `.env` 管理，与 acore-ranking 的配置需求一致 |
| **项目结构参考** | 模块化目录结构（`auth/`、`characters/`、`world/`、`shared/`）可作为 acore-ranking 后端工程组织的参考 |

**注意**：`acore-api` 的定位是"服务端管理 API"（GM 工具、角色服务），而 `acore-ranking` 的定位是"数据展示面板"（排行榜、统计）。两者互补而非替代——acore-ranking 可直接复用其 Entity 和数据库连接模式，但排行榜查询逻辑、前端展示层需独立实现。
